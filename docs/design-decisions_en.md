# Architecture Decision Record

This document records the key technical decisions made during the system design phase along with the trade-off considerations behind them. It complements [`project-architecture-and-introduction.md`](project-architecture-and-introduction.md)—the architecture document describes "what the system is," while this document explains "why it is designed this way."

---

## 1. Use the OpenClaw CLI Instead of the HTTP API

**Decision**: All interaction with OpenClaw goes through CLI subprocesses (`openclaw agent --message ...`), rather than the HTTP API.

**Rationale**:
- The CLI is the official interface that OpenClaw exposes externally; its behavior is consistent with what users actually experience
- The HTTP API is the Gateway's internal interface, whose paths and formats are prone to change across versions
- Driving through the CLI means the tool list and system prompt are generated by the real runtime, rather than being statically defined

---

## 2. Separation of Responsibilities Between Two Models

**Decision**: OpenClaw's underlying execution model and the external LLMClient (the user simulator) are completely independent and do not interfere with each other.

**Rationale**:
- The execution model decides "how to do the task" (tool call strategy, reasoning depth) and can be any model that supports tool calls
- The user simulator decides "what the user says" (query decomposition, completion judgment); it has high demands on task understanding but no requirements on tool calls
- After separation, the two sets of models can be replaced independently (for example, Claude for execution and Qwen for simulation) without constraining each other

---

## 3. Runtime Probe Instead of Static Tool Scanning

**Decision**: Capture the `tools` + `system_prompt` that OpenClaw actually sends out to the model through a short-lived probe, rather than statically scanning source code or configuration files.

**Rationale**:
- At runtime, OpenClaw dynamically assembles the tool list based on installed plugins, enabled skills, and agent configuration
- Static scanning can only obtain "tools that might exist," not "the tool definitions this agent actually receives"
- A runtime probe captures the real context from the model's perspective, so the `tools` field in the training data is trustworthy

---

## 4. Deferred Session Finalization

**Decision**: The archiving and format conversion of a session is performed only after the User Loop has fully completed, rather than being written turn by turn.

**Rationale**:
- An OpenClaw session file keeps appending content while the agent continues interacting; reading it too early yields an incomplete session
- Deferring to a one-time archive after the final state guarantees that the session file is complete and replayable
- A failed session has a complete pending record, so after a restart it can resume from the breakpoint rather than rerunning from scratch

---

## 5. Thread Pool Concurrency Instead of Multiple Processes

**Decision**: Concurrency uses `ThreadPoolExecutor`, rather than `ProcessPoolExecutor` or `asyncio`.

**Rationale**:
- A worker spends most of its time waiting on the OpenClaw CLI subprocess and LLM API responses—all I/O blocking, so the GIL is not the bottleneck
- Threads can directly share objects such as `shared_llm`, `shared_converter`, and `shared_tools` without IPC
- Multiple processes would require serializing shared objects and establishing inter-process channels, adding complexity without any CPU compute benefit

---

## 6. JSON for the Progress File Instead of a Database

**Decision**: Task progress is recorded in `output/progress.json`, rather than SQLite or another database.

**Rationale**:
- The data volume is limited (thousands of intents), so JSON is efficient enough
- The file can be inspected and manually repaired directly with text tools, keeping debugging costs low
- No extra dependencies; multi-worker writes are protected by `threading.Lock()`, which is sufficient to guarantee consistency

---

## 7. Configuration Priority: ENV > CLI > config.yaml

**Decision**: All configuration items support three layers of overriding, with environment variables having the highest priority.

**Rationale**:
- Injecting runtime parameters via `-e` during container deployment is standard practice, with no need to mount a modified configuration file
- CLI arguments make it convenient to quickly override a single value during local debugging
- config.yaml stores the stable baseline configuration and is version-control friendly
